The Problem Is Unintended Systemic Bias


Existing technology is systemically denying blind screen reader users an equal user experience. Because summary information about Web pages is being withheld from blind assistive technology users. Conversely, robots have had access to summary information in the form of meta description tags, open graph tags and SEO markup for many years. While sighted Web page users can often determine when a Web page is of interest to them in under three seconds. Yet blind screen reader users have been expected to tab from heading to heading and alt text to alt text, literally flying blind through HTML, in order to determine when a Web page is of interest to them. To make blind assistive technology users equal to sited users and robots, I'm proposing the adoption of Accessible Page Summaries. Because including a brief summary, along with information about content size, will allow blind screen reader users to finally make quick determinations about Web pages, just like sited users and robots have been able to do for a long time.

Why not rely on open graph and meta description tags for summarized content? Because the needs of search engines and the needs of blind assistive technology users are fundamentaly different. Basically, SEO is a methodical reaction to every Web page jockying for position on Search Engine Results Pages. Which is why open graph and meta description tags attempt to include every relevant search term, in order to increase the odds of click thoughs. What blind screen reader users need is not a competition for their attention. They need document specific summaries that work with no dependencies, that get right to the point, with content composed specifically for humans, and valid informatin about content length.

The Solution:


Introducing Accessible Page Summaries, an innovative way to provide content size and high quality preliminary information (who, what, where, when and why) intended specifically to serve the needs of blind assistive technology users.

▶ Everyone still gets the same content.
▶ Robots are currently getting preliminary information that sighted visitors do not need.
▶ Screen reader users need better summary information than robots are now getting.
▶ Accessible Page Summaries give screen reader users high quality preliminary information to meet their needs.

NOTE: This solution has no dependencies so it is always accessible even when CDN's fail. Being integrated into each Web page means it is never subject to URL and redirect errors. By being an integral part of each Web page it will always provide the correct content length and summary information, no matter what.

How It Works:

By design, the Accessible Page Summary is positioned so far to one side that no one can see it and it will not change the appearance of any Web page. When blind assistive technology users visit a Web page with an Accessible Page Summary, their screen reader software gives them the option to open it and access the preliminary content, or by default, continue past it when revisiting a Web page.

Risks and Mitigation:

The included CSS does three things. 1. It provides functionality to open the Accessible Page Summary. 2. It positions the Accessible Page Summary off screen. 3. It formats the Accessible Page Summary to pass WCAG 2.1 AAA.

Because CSS is fundamental to HTML, CSS was chosen over Java script, to handle the opt-in "open" function that can be selected by screen reader users.

Here is why the CSS must be included in the head section of every Web page. Because when the functionality of the CSS is sourced from a dependency CSS file it may not function offline, may not function due to CDN failure, may fail due to redirect errors and may fail as a result of DNS errors. By being included in the head section, and by being 100% free of dependencies, the CSS open function cannot fail, the summary will always remain off screen, and will be WCAG 2.1 AAA compliant at all times.

What Happens Next?

The examples below are for Web pages. More work is needed to develop Accessible Page Summaries for PDF's, Ebooks, and other document types. It is my hope that talented engineers will be inspired by this open source repository and make it happen.

The Accessible Page Summary should be a WCAG requirement. It might also make sense to include the Accessible Page Summary as fundamental to HTML5.

What Else Can It Do?

Future versions could be opened by an "i" for information button, to be included in every Web browser, intended for sited users, that would open complete Accessible Page Summary content in a sidebar. Then everyone will be able to benefit from Accessible Page Summaries.

For All Versions.


    
<!-- Accessible Page Summary CSS - BEGIN -->
<style>/* Opt-Out (default) */.ADA-ps {display: none}/* Opt-In (selectable) */.Summary-Opt-In:focus+.ADA-ps {display: block}/* WCAG 2.1 AAA */.ada-ps-css-01 {background: #fff; color: #000;display: inline-block; font-size: 1.5rem; line-height: 150%; margin-left: -3000rem; position: absolute; z-index: 997}</style>
<!-- Accessible Page Summary CSS - END -->

ADA P.S. 1

HTML For typical Web pages.


    
<!-- FOR TYPICAL WEB PAGES - ADA P.S. 1 -->
<!-- Accessible Page Summary - BEGIN -->
<!-- Passed WCAG 2.1 AAA 08/25/2023 -->
    <section role="complementary" aria-label="Note about usage" class="ada-ps-css-01">
    <h2>To Hear a Brief Summary of This Web Page Including Content Size Click Open. Then use the down arrow.</h2>
    <button aria-pressed="false" value="Open" class="Summary-Opt-In">Open</button>
    <section role="complementary" aria-label="Note about usage" class="ADA-ps">
    <p><!-- description -->
    Web Page Description.<br>
    Replace this line with a Description of the Web page that does not duplicate the meta description tag.</p>
    <p><!-- alt text summary -->
    Alt Text Summary.<br>
    Replace this line with a very brief summation of all images.</p>
    <p><!-- content size -->
    Content Size.<br>
    Replace this line with the word count or, short reed, long reed, length of video, etc.
    <!-- never remove this line -->
    That completes the Summary. The Web page content starts now.</p>
    </section></section>
<!-- Accessible Page Summary - END -->

The above source code is for typical Web pages that have text and images.

INSTALLATION AND SETUP

Always make a backup copy of a Web page before making any changes to it.

How to Install

NOTE: This installation cannot change the appearance of a Web page.

1. Insert the CSS anywhere within the head section of the Web page. Do not add the CSS to a CSS file. Integrate the CSS directly into each Web page so it will never fail.

2. Within the body section, insert the HTML directly at the end of the H1 tag. This is the required location in order for the summary to function. Please observe the following example.

Example: <h1>Hello World!</h1><!-- FOR TYPICAL WEB PAGES - ADA P.S. 1 -->

How to Fill Out

Keep the description brief. EG: What, when, where, who and why. Is there a how-to? If yes, don't explain how-to's in the description. Just let readers know that instructions are included. End with content size.

Take advantage of the alt text summary. Because this is a composite of all the images, it can tell a very brief story. For example: The images on this Web page show people fertilizing soil, planting seeds, watering a garden, growing vegetables, harvesting vegetables, and eventually enjoying a feast.

Content size can be a precise word count or an approximate word count EG: less than 2000 words. Content size can also be the running time of a video or music, or it can be as simple as: short read, long read, very long read.

About The Project:
The idea to create an accessible page summary resulted from a search to determine how many blind people use the internet. Very few. The search result was surprising. Because the blind have a strong preference for braille libraries over the internet. Not surprisingly, the most popular feature at braille libraries is the card catalog. Because every card in the catalog provides a brief summary of every book and includes details about content size. If only the World Wide Web had a card catalog. When every Web page has an Accessible Page Summary it will.

About The Developer:
It is essential to know why the developer is so passionate about this project. Because Stuart M. Long's late brother Austin was a beneficiary of the Americans with Disabilities Act as-well-as a card carrying member of MENSA. While in a one-on-one business meeting, Long had to suffer through an insensitive sociopath's rant about the challenges involved in complying with the ADA. That individual made it seem as though doing the right thing and helping others was a burden. After explaining that his late brother lived with a disability every day of his adult life, and the ADA was a dream come true, the sociopath slowly took a deep breath and then apologized. The memory of that meeting drove home a very fine point, that some people will never understand what it is like to live with a disability.

Relevant Background:
When working for the Kobacker Company, formerly based out of Columbus Ohio, Long demonstrated a gift for working with business computers. Because out of over 700 locations, Long devised innovative methods and techniques to process payments that made him the fastest and most accurate at completing transactions company wide. His methods and techniques were added to the operations manual and became standard procedure for all employees. He also traveled out-of-town to various locations to setup and configure business machines and networks.

Stuart M. Long describes himself as a standards compliance enthusiast. Because he wants assistive technology to work properly for everyone. This is Long's inaugural Github project.

Personal note from Stuart M. Long:
"One morning, I put new ski shaped feet on the back legs of my mom's walker. Then I watched her go for a walk and she loved them. Because they helped her get around with less friction and made her walk more enjoyable. That's what I want the Accessible Page Summary to do for screen reader users; reduce friction and make surfing the Web more enjoyable."
